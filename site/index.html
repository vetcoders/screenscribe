THE PROBLEM

A recording isn't a bug report

Hitting record and talking through what's broken is the fastest way to show it. But a video isn't actionable - someone still has to watch the whole thing, scrub to the right second, and type up what was said. That someone is usually future-you, at the worst possible time.

HOW IT WORKS

from 'watch this' to 'fix this'

One narrated screencast in, a structured report out. Three stages do the watching, scrubbing, and writing for you.

narrationyou, talking through the bug
→
STTspeech-to-text
turns your talking into text
→
LLMlanguage model
reads it, finds what matters
→
VLMvision model
checks it against the frame
→
reportJSON · Markdown · HTML

Each stage passes its context to the next - so the vision check already knows what you said.

01

Record

Capture your screen and narrate the bugs and changes in plain language. Any format FFmpeg reads works - mp4, mov, mkv, webm.

02

Pipeline

screenscribe transcribes the narration, finds the moments that matter, then confirms each one against the actual frame on screen.

03

Report

Findings with timestamps, screenshots, and a summary - as JSON, Markdown, and one interactive HTML file you can just open.

INSTALL

Get started in three steps

Clone and install, add your key, run a review. Bring your own OpenAI-compatible key - you pay the provider directly, nothing in between.

1

Clone & install

Grab the repo and run make install. It installs the CLI as a bare screenscribe command via uv tool.

2

Add your key

Initialize config and drop in your OpenAI-compatible key.

3

Run a review

Point it at a recording - the report opens in your browser.

BASH 
$ git clone https://github.com/vetcoders/screenscribe.git
$ cd screenscribe
$ make install
# add your OpenAI-compatible key
$ screenscribe config --init
$ screenscribe config --set-key YOUR_API_KEY
# review a narrated recording
$ screenscribe review demo.mov
Python 3.11+ FFmpeg uv make
